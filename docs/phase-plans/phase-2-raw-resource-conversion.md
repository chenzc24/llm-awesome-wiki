# Phase 2 Raw Resource To Source Packet Protocol

Phase 2 defines the protocol for turning raw resources into source packets.

It does not define a MinerU-style document parsing harness. It does not require
one extractor, one model backend, one MCP server, or one PDF/PPTX/DOCX parser.
Instead, it defines the repo-local contract that any human, agent, MCP, local
CLI, or external extractor must satisfy before later evidence, claims, wiki
pages, and raw-wiki alignment checks can trust the source side.

The file name still says `raw-resource-conversion` for continuity, but the
phase meaning is protocol-first:

```text
raw resource
-> source inventory
-> source packet protocol
-> anchor references
-> future wiki alignment checks
-> evidence/claims
-> wiki
```

## Position In The System

Phase 1 made the workspace kernel copyable and manually usable.

Phase 2 starts the first construction-tool layer and establishes the raw side
of raw-wiki alignment. It stops before claim extraction and wiki generation.
It does not build compare gates or downstream `skill + tool` artifacts.

Phase 2 must define enough source identity, packet structure, anchor discipline,
generated-field marking, and gap reporting that later tools can check coverage
without relying on model memory or self-evaluation.

## Phase Thesis

The system should make raw resources auditable before wiki writing begins,
while avoiding audit artifacts that make the later agent-readable wiki harder
to maintain.

A source packet is not just extracted text. It is a traceable bridge that says:

- which raw file was processed
- which method or backend produced each extracted artifact
- which anchors can be cited later
- which modalities were present
- which parts failed, were skipped, or need review
- which fields were generated by OCR, VLM, an agent, or another model-assisted
  process instead of directly extracted from the source

This keeps the project Agent-first and Git-first: an agent can reason over
repo-local files, while the repository records enough evidence for humans and
deterministic tools to audit what happened.

Artifact economy applies here. A source packet is the source of truth for
anchors, extraction methods, generated fields, and extraction gaps; it is not a
second wiki article. Later wiki pages should cite packet anchors instead of
copying raw hashes, backend metadata, and extraction logs into the
agent-readable knowledge surface.

## What MinerU Contributes

MinerU is useful as reference material because it treats document parsing as a
multi-layer output problem rather than a single Markdown dump.

The patterns worth borrowing are output and audit patterns:

- **Backend declaration**: outputs record whether parsing came from a pipeline,
  VLM, hybrid mode, office parser, or HTTP client.
- **Page-oriented anchors**: parsed blocks can carry page numbers and bounding
  boxes that later evidence references may cite.
- **Structured intermediate files**: detailed parser/model output is separated
  from simplified downstream content.
- **Reading-order content lists**: simplified block lists are easier for later
  tools than arbitrary Markdown.
- **Visual debug artifacts**: layout and span artifacts help reviewers inspect
  extraction quality.
- **Modality-specific blocks**: text, title, table, chart, image, formula,
  code, list, header, footer, footnote, and page number blocks are separated.
- **Partial and complex layout awareness**: scanned pages, visual-heavy pages,
  formulas, charts, and tables are treated as first-class extraction concerns.

These ideas should influence our packet protocol, report language, and future
CLI specs.

## What We Do Not Inherit From MinerU

This project should not inherit MinerU's engineering shape.

Do not make Phase 2 depend on:

- a PDF/PPTX/DOCX harness implemented in this repository
- a GPU being present
- a local model server
- a hosted API
- a desktop client
- a WebUI
- MCP integration
- a long-running router service
- MinerU-specific output paths as our source of truth

MinerU can be one optional extractor backend later. Claude Code MCPs, Codex
local tools, Poppler, LibreOffice, manual extraction, and custom scripts can
also be backends. The workspace protocol remains the user-facing source of
truth.

The default path must remain CPU-first and deterministic where possible. OCR,
VLM captioning, chart interpretation, image understanding, and model-assisted
cleanup are optional enhancement layers. When those layers are used, their
outputs must be marked as generated or model-assisted and routed to review when
they affect claims.

## Agent And Extractor Relationship

The protocol should guide agents without assuming a specific agent runtime.

For example:

```text
Codex / Claude Code / Cursor / local CLI / MCP / MinerU
= optional executor or extractor

source inventory / source packet / metadata / anchors / gaps
= required workspace protocol
```

An agent with its own MCP may choose how to read a PDF or PPTX internally. This
project defines what it must leave behind:

- inventory row
- stable source ID
- packet path
- extraction method and version
- anchors
- generated-field markers
- known gaps
- review notes
- derived artifact references when useful

## Proposed Work Breakdown

### Phase 2.1 Source Intake And Inventory

Status: complete.

Define the first step after a user copies files into `raw/sources/`.

The inventory workflow specifies:

- source ID creation
- raw path conventions
- hashing expectations
- source kind detection
- status values such as `new`, `ready`, `processed`, `changed`, `ignored`,
  `failed`, and `needs-review`
- when inventory is manual versus tool-generated
- how ignored or unsupported files are recorded

MinerU reference point: the input path can be a file or directory, but our
workspace normalizes that into stable source IDs and inventory rows before any
extractor backend is chosen.

### Phase 2.2 Source Packet Protocol

Status: complete.

Define the minimum packet that every extractor, agent, MCP, or manual workflow
must produce.

The packet protocol should specify:

- required packet files, such as `source.md`, metadata, media references, and
  optional review notes
- source identity fields inherited from inventory
- extraction method, backend, version, and runtime notes
- source structure summary
- anchor shape and naming
- extracted content blocks
- generated fields
- extraction gaps
- review routing
- handoff notes for evidence/claim and wiki stages

MinerU reference point: keep simplified downstream content separate from
detailed intermediate/debug data. A packet can link to detailed derived
artifacts without forcing later phases to parse backend-specific files.

### Phase 2.3 Extractor Adapter Protocol

Status: complete.

Define how optional extractors plug into the workspace without becoming the
workspace owner.

Adapter protocol should specify:

- accepted inputs from inventory and raw sources
- required packet outputs
- how to declare backend name, method, version, and configuration
- how to map backend-specific anchors into workspace anchors
- how to reference backend-specific artifacts without making them source
  identity
- how to mark unsupported modalities and partial output
- how to fail visibly

Examples of possible adapters:

- manual extraction
- Poppler-style PDF text/render tools
- LibreOffice-style office conversion
- MinerU
- Claude Code MCP
- Codex local tools
- custom domain scripts

Do not implement adapter code in this planning phase.

### Phase 2.4 Source-Type Packet Profiles

Status: current target.

Define source-type minimum requirements for packet outputs. These are not
parser implementations.

PDF packet profile:

- page-level anchors
- page count and page range tracking
- text extraction result or explicit text-extraction gap
- rendered page references for visual-heavy pages when available
- empty, scanned, password-protected, or extraction-failed pages recorded
- table/chart/image blocks as separate anchors where possible
- generated captions and chart interpretations marked separately from source
  text

PPTX packet profile:

- slide-level anchors
- slide order preservation
- slide text extraction result or explicit gap
- speaker notes extraction when available
- rendered slide references for layout-heavy slides when available
- uncertain media-to-slide mapping recorded as review

DOCX packet profile:

- heading anchors
- heading hierarchy preservation
- paragraph, list, table, footnote, and comment extraction when relevant
- small tables as Markdown when feasible
- large tables linked as companion CSV/TSV or derived artifacts
- meaningful images extracted, referenced, or routed to review

Image/table/mixed-media profile:

- distinguish extracted data from model-generated captions
- record whether captions are reviewed
- route unsupported visual interpretation to review
- preserve derived artifact references without pretending they are direct
  source text

### Phase 2.5 Generated Fields And Review Routing

Define how the workflow distinguishes source-derived content from generated or
model-assisted content.

Generated fields may include:

- OCR text
- image captions
- chart summaries
- table repairs
- formula recognition
- inferred reading order
- agent-written extraction notes

Rules should state:

- generated content is allowed inside source packets
- generated content must be labeled
- generated content is not source-equivalent by default
- claims based on generated content require evidence flags or review
- unsupported modalities should create review items instead of disappearing

MinerU reference point: VLM and hybrid output can be useful, but the workflow
must not let model output silently become trusted source text.

### Phase 2.6 Tool Surface Specs

Update README-level tool specifications before implementation.

Candidate tools:

- `source-inventory`: scan `raw/sources/` and write inventory rows or reports
- `raw-resource-convert`: run an adapter or manual workflow and emit a source
  packet
- `source-packet-lint`: check anchors, metadata, generated fields, gaps, and
  review routing

The specs should define:

- command purpose
- inputs
- outputs
- report paths
- failure modes
- deterministic behavior
- optional extractor backend behavior
- exit-code expectations when known

Do not implement these tools in the planning pass.

## Non-Goals

Phase 2 should not:

- implement a complete raw converter
- implement a PDF/PPTX/DOCX parsing harness
- run MinerU as a required dependency
- require GPU or model service access
- require any MCP server
- generate final wiki pages
- extract claims
- implement compare gates
- add active root-level `raw/`, `wiki/`, `reports/`, or `schema.md`
- start downstream skill/tool generation

## Completion Criteria

Phase 2 is complete when:

- `raw-to-source-packet` rules describe the full source packet protocol
- source packet templates show required files, anchors, metadata, generated
  fields, gaps, and review routing
- extractor adapter protocol explains how optional tools/MCPs/backends plug in
  without owning the workspace
- source-type packet profiles exist for PDF, PPTX, DOCX, images, tables, and
  mixed media
- Phase 2 artifacts make future raw-wiki alignment reports possible without
  making source packets the wiki knowledge surface
- tool README specs explain the future CLI surfaces without overpromising
  implementation
- Person A has a clear schema/validator handoff list
- the default path remains CPU-first and repo-local
- optional MinerU or VLM usage is described as backend input, not workflow
  ownership
