# Phase 2 Raw Resource Conversion

Phase 2 defines the workflow surface for converting raw resources into source
packets. It is also the first step of the raw-wiki alignment substrate: the
source identities, anchors, generated-field markers, and gaps created here are
what later wiki pages and alignment reports must cite.

The phase is owned by Person B on the workflow/tool-surface line. It uses
`MinerU/` as read-only reference material for document parsing patterns, but it
does not turn this project into a MinerU wrapper, an Obsidian vault, a desktop
application, or a model-serving platform.

## Position In The System

Phase 1 made the workspace kernel copyable and manually usable.

Phase 2 starts the first construction-tool layer and establishes the raw side
of raw-wiki alignment:

```text
raw resource
-> source inventory
-> source packet
-> anchor references
-> future wiki alignment checks
-> evidence/claims
-> wiki
```

Phase 2 stops before claim extraction and wiki generation. It does not build
compare gates or downstream `skill + tool` artifacts. But it must define enough
source identity and anchor discipline that later raw-wiki alignment is
checkable instead of relying on model memory.

## Phase Thesis

The system should make raw resources auditable before any wiki writing begins,
while avoiding audit artifacts that make the later readable wiki harder to
maintain.

A source packet is not just extracted text. It is a traceable bridge that says:

- which raw file was processed
- which method produced each extracted artifact
- which anchors can be cited later
- which modalities were present
- which parts failed, were skipped, or need review
- which fields were generated by OCR, VLM, or an agent instead of directly
  extracted from the source

This keeps the project Agent-first and Git-first: an agent can reason over
repo-local files, but the repository still records enough evidence for a human
or deterministic tool to audit what happened.

Artifact economy applies here: a source packet is the source of truth for
anchors and extraction gaps, not a second wiki article. Later wiki pages should
cite packet anchors instead of copying packet metadata and extraction logs.

## What MinerU Contributes

MinerU is useful because it treats document parsing as a multi-layer output
problem rather than a single Markdown dump.

The main patterns worth borrowing are:

- **Backend declaration**: outputs record whether parsing came from a pipeline,
  VLM, hybrid mode, office parser, or HTTP client.
- **Page-oriented anchors**: parsed blocks carry page numbers and bounding
  boxes, which are useful for later evidence references.
- **Structured intermediate files**: `middle.json`, `model.json`, and content
  lists separate raw model/parser output from simplified downstream content.
- **Reading-order content lists**: simplified block lists are easier for later
  tools than arbitrary Markdown.
- **Visual debug artifacts**: layout and span PDFs help reviewers inspect
  whether extraction preserved order and coverage.
- **Modality-specific blocks**: text, title, table, chart, image, formula,
  code, list, header, footer, footnote, and page number blocks are separated.
- **Partial and complex layout awareness**: scanned pages, visual-heavy pages,
  formulas, charts, and tables are treated as first-class extraction concerns.

These ideas should influence our source packet format, report language, and
future CLI specs.

## What We Do Not Inherit From MinerU

This project should not inherit MinerU's product shape.

Do not make Phase 2 depend on:

- a GPU being present
- a local model server
- a hosted API
- a desktop client
- a WebUI
- MCP integration
- a long-running router service
- MinerU-specific output paths as our source of truth

MinerU can be one extractor backend later. The workspace kernel remains the
user-facing surface.

The default path must remain CPU-first and deterministic where possible. OCR,
VLM captioning, chart interpretation, and image understanding are optional
enhancement layers. When those layers are used, their outputs must be marked as
generated or model-assisted and routed to review when they affect claims.

Phase 2 should produce workflow and tool-surface clarity in Person B-owned

Expected deliverables:

- refined `rules/raw-to-source-packet.md`

## Proposed Work Breakdown

### Phase 2.1 Source Intake And Inventory

Define the first step after a user copies files into `raw/sources/`.

The inventory workflow should specify:

- source ID creation
- raw path conventions
- hashing expectations
- source kind detection
- status values: `new`, `processed`, `changed`, `ignored`, `failed`
- when inventory is manual versus tool-generated
- how ignored or unsupported files are recorded

MinerU reference point: the input path can be a file or directory, but our
workspace should always normalize that into stable source IDs and inventory
rows before extraction.

### Phase 2.2 Source Packet Shape

Define the minimum packet that every extractor or agent must produce.

The packet should separate:

- source identity
- extraction method and version
- source structure
- extracted content
- media references
- generated fields
- extraction gaps
- review items
- handoff notes

MinerU reference point: keep simplified downstream content separate from
detailed intermediate/debug data. Our packet can link to detailed derived
artifacts without forcing every later phase to parse them.

### Phase 2.3 PDF Profile

PDF is the first serious target because the ADCtoolbox dry run exposed the real
shape of the problem: page text can be easy, but charts, layouts, and visual
evidence need stricter handling.

The PDF profile should require:

- page-level anchors
- page count and page range tracking
- text extraction before model analysis
- rendered page images for visual-heavy pages
- explicit records for empty, scanned, password-protected, or failed pages
- table/chart/image blocks as separate anchors where possible
- generated captions and chart interpretations marked separately from source
  text

MinerU reference point: page index, bounding boxes, layout PDFs, span PDFs, and
content block types are the useful ideas. GPU-backed VLM parsing is optional,
not baseline.

### Phase 2.4 PPTX And DOCX Profiles

PPTX and DOCX need source-order preservation more than aggressive research-wiki
object extraction.

The PPTX profile should require:

- slide-level anchors
- slide order preservation
- slide text extraction
- speaker notes extraction when available
- rendered slide screenshots for layout-heavy slides
- uncertain media-to-slide mapping recorded as review

The DOCX profile should require:

- heading hierarchy preservation
- paragraph, list, table, footnote, and comment extraction when relevant
- small tables converted to Markdown
- large tables linked as companion CSV/TSV files
- meaningful images extracted or referenced

MinerU reference point: native office parsing is valuable, but our workflow
should keep the source packet stable even when the backend changes.

### Phase 2.5 Generated Fields And Review Routing

Define how the workflow distinguishes source-derived content from generated or
model-assisted content.

Generated fields may include:

- OCR text
- image captions
- chart summaries
- table repairs
- formula recognition
- inferred reading order
- agent-written extraction notes

Rules should state:

- generated content is allowed inside source packets
- generated content must be labeled
- generated content is not source-equivalent by default
- claims based on generated content require evidence flags or review
- unsupported modalities should create review items instead of disappearing

MinerU reference point: VLM and hybrid output can be useful, but the workflow
must not let model output silently become trusted source text.

### Phase 2.6 Tool Surface Specs

Update README-level tool specifications before implementation.

Candidate tools:

- `source-inventory`: scan `raw/sources/` and write inventory rows or reports
- `raw-resource-convert`: convert one raw source into a source packet
- `source-packet-lint`: check anchors, metadata, generated fields, and gaps

The specs should define:

- command purpose
- inputs
- outputs
- report paths
- failure modes
- deterministic behavior
- optional extractor backend behavior
- exit-code expectations when known

Do not implement these tools in the planning pass.

## Non-Goals

Phase 2 should not:

- implement a complete raw converter
- run MinerU as a required dependency
- require GPU or model service access
- generate final wiki pages
- extract claims
- implement compare gates
- add active root-level `raw/`, `wiki/`, `reports/`, or `schema.md`
- start downstream skill/tool generation

## Completion Criteria

Phase 2 is complete when:

- `raw-to-source-packet` rules describe the full source packet workflow
- modality profiles exist for PDF, PPTX, DOCX, images, tables, and mixed media
- workspace templates show how to record source inventory, packet anchors,
  generated fields, gaps, and review routing
- Phase 2 artifacts make future raw-wiki alignment reports possible without
  making source packets the human reading surface
- tool README specs explain the future CLI surfaces without overpromising
  implementation
- Person A has a clear schema/validator handoff list
- the default path remains CPU-first and repo-local
- optional MinerU or VLM usage is described as backend input, not workflow
  ownership
