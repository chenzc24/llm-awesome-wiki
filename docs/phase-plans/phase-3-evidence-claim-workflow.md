# Phase 3 Evidence And Claim Workflow

Phase 3 defines the Person B workflow surface for turning Phase 2 source
packets into traceable evidence and claim records.

This phase is not raw conversion and not wiki generation. It consumes source
packet anchors and prepares the audit layer that later wiki construction and
compare gates can use.

## Position In The Pipeline

```text
raw source
-> source inventory
-> source packet
-> evidence and claim layer
-> source/chapter-oriented wiki
-> compare and review gates
-> stable release/spec layer
-> later downstream skill + tool artifacts
```

Phase 2 owns the path up to a valid source packet. Phase 3 begins after a
packet has stable anchors such as `<source_id>#<anchor_id>`.

## Core Boundary

The evidence and claim layer is an audit and review surface. It is not the
primary reading surface.

For document and PPT corpora, the default human/agent reading path remains:

```text
wiki/overview.md
-> wiki/index.md
-> wiki/chapters/*
-> optional short source pages
```

Evidence and claim artifacts support that path by answering:

- which anchors support important knowledge
- which claims are source-derived, generated-derived, contested, or weak
- which uncertainties must be routed to human review
- which claims are ready for later wiki pages and compare gates

Do not turn the default corpus into a fragmented concept/entity/research-object
graph just because claim records exist.

## Phase 3 Outputs

Person B should produce workflow surfaces for:

- source packet anchor to evidence selection
- evidence to claim extraction
- generated-evidence warning and review routing
- claim/evidence report templates
- future claim-audit tool behavior prose
- Person A handoff notes for schema and validator hardening

The preferred workspace artifact in this phase is a concise report or map, for
example:

```text
reports/review/claim-evidence-map.md
reports/review/review-queue.md
```

These reports can later be replaced or generated by tools. They should not
become a second wiki or a duplicate source packet.

## Evidence Role

Evidence is a cited support unit for a claim.

Minimum evidence workflow facts:

- source packet read
- source reference in `<source_id>#<anchor_id>` form
- evidence kind, such as text, table, figure, OCR, caption, formula, summary,
  or inference
- generated state when the evidence depends on OCR, VLM output, agent notes,
  inferred ordering, repaired tables, or other generated fields
- confidence or review need when the support is weak, generated, ambiguous, or
  incomplete

Evidence should stay close to source packet anchors. It should not repeat all
packet metadata, raw hashes, or extraction internals unless those facts affect
review.

## Claim Role

A claim is a reusable knowledge statement important enough to preserve,
compare, review, or cite.

Not every sentence is a claim. A statement may remain ordinary chapter context
when it is only:

- a heading or navigation label
- an overview of what a section covers
- an extraction note
- a temporary summary
- a reading aid that is not meant to be reused as knowledge

Claims are useful when they carry one of these roles:

- definition
- procedure
- design rule
- comparison
- numerical or parameter statement
- causal or mechanism statement
- limitation, warning, or exception
- requirement or acceptance criterion
- open question that blocks downstream use

## Status Meanings

Use these workflow status meanings consistently:

| Status | Meaning |
| --- | --- |
| `draft` | Candidate claim exists but support has not been checked. |
| `supported` | Evidence anchors support the claim for the current scope. |
| `contested` | Sources conflict or the claim is disputed by another source. |
| `unsupported` | Claim lacks sufficient evidence and should not be promoted. |
| `needs-review` | Deterministic checks cannot decide or human judgment is required. |
| `stale` | Source material, packet hash, or review decision changed after claim creation. |

These are workflow meanings. Person A owns final schema validation and enum
hardening.

## Generated Evidence

Generated evidence includes OCR text, image captions, chart summaries, table
repairs, formula recognition, inferred reading order, visual descriptions,
agent extraction notes, normalized values, inferred headings, and cross-file
summaries.

Generated evidence is allowed, but it must remain visible. Important claims
based on generated evidence must either:

- carry the generated-evidence state into the claim/evidence map
- be marked `needs-review`
- cite a review decision that accepted the generated evidence

Do not present unreviewed generated evidence as source-derived knowledge.

## Review Routing

An agent should create or update a review item instead of deciding alone when:

- an important claim depends on generated evidence
- an anchor does not fully support the claim
- sources conflict
- a numerical value, formula, procedure, or design rule is uncertain
- evidence is missing because a modality was skipped
- the claim is broader than the source packet supports
- human judgment is needed to resolve scope, meaning, or acceptance

Review items should name the claim, source references, generated fields if any,
and the exact decision needed.

## Artifact Economy Rules

Phase 3 must preserve artifact economy:

- source packets remain the audit source of packet anchors
- claim/evidence maps record support relationships, not full packet content
- wiki chapters remain the primary agent-readable knowledge layer
- review queues record unresolved judgment, not polished explanations
- logs record events, not knowledge summaries

If a claim/evidence artifact duplicates a fact, it must either cite the source
of truth or explain why the duplication is needed for review.

## Person A Handoff

Person B should not edit `contracts/schemas/**` in this phase. Handoff needs
for Person A include:

- evidence references should support `<source_id>#<anchor_id>`
- claim records may need `claim_type`
- claim records may need generated-evidence or provenance markers
- review items may need `evidence_ids`
- validators should catch `supported` claims with no evidence
- validators should route generated-evidence claims to review unless a review
  decision exists

## Non-Goals

Phase 3 does not:

- reopen raw conversion or source packet generation
- implement parser, extractor, MCP, CLI, schema, validator, or fixture code
- generate final wiki pages
- implement compare gates
- require Obsidian or any desktop app
- create downstream `skill + tool` artifacts

## Acceptance Criteria

Phase 3 is complete when:

- rules explain source packet to evidence selection
- rules explain evidence to claim extraction
- review routing for generated and uncertain evidence is explicit
- templates show a lightweight claim/evidence map and review queue
- tool prose explains future claim-audit behavior without implementing it
- closure review confirms alignment with core design philosophy
- Person A handoff needs are recorded without schema edits
