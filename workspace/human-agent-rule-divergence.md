# llm-wiki 中“给人读”和“给 Agent 执行”的分歧

## 背景

`llm-wiki.md` 当前更像一篇理念说明文，目标是向人解释一种使用 LLM 维护个人知识库的模式。它能帮助读者理解这个想法为什么有价值，但不完全适合作为 Agent 的直接执行规则。

如果要让 Codex、Claude Code 或其他 LLM Agent 真正维护 wiki，需要把这类说明文进一步改写成更明确、可操作、可检查的规则文件，例如 `AGENTS.md` 或 `CLAUDE.md`。

## 核心分歧

给人读的文档需要解释“为什么这样做”。它会包含背景、类比、例子、动机、适用场景和说服性表达。

给 Agent 执行的文档需要说明“必须怎么做”。它应该包含目录结构、文件职责、页面格式、处理流程、更新规则、引用规则和检查清单。

因此，当前 `llm-wiki.md` 的问题不是内容错误，而是用途偏向人类理解，不够偏向机器执行。

## 当前文档中偏向人读的内容

- 反复解释 RAG 的局限，以及为什么 persistent wiki 更好。
- 使用较多理念性表达，例如知识会积累、wiki 会 compounding。
- 包含类比和叙述，例如 Obsidian 是 IDE、LLM 是程序员、wiki 是代码库。
- 列举多种适用场景，例如个人成长、研究、读书、团队知识库、竞争分析等。
- 较长地介绍可选工具，例如 Obsidian、qmd、Marp、Dataview、Web Clipper。
- 结尾包含 Memex 等思想来源和哲学性说明。

这些内容对人有帮助，但对 Agent 的日常执行帮助有限。

## Agent 规则真正需要保留的内容

- 原始资料是 source of truth，Agent 只能读取，不应修改。
- wiki 是 Agent 维护的 Markdown 层，负责承载总结、实体、概念、分析和综合结论。
- schema 文件负责规定 Agent 的行为规范。
- 新资料进入时要执行 ingest，不只是写单篇摘要，还要更新相关页面。
- 回答问题时应优先查 `index.md`，再读取相关 wiki 页面。
- 有价值的回答、比较、分析和洞见可以沉淀回 wiki。
- `index.md` 是内容索引，应在每次重要更新后维护。
- `log.md` 是时间日志，应记录 ingest、query、lint 等操作。
- 定期 lint，检查矛盾、过期结论、孤立页面、缺失链接、缺失概念页和资料空白。

## 适合压缩或移出的内容

- RAG 与 wiki 的长篇对比可以压缩成一两条原则。
- 适用场景可以移到 README 或介绍文档，不必放进 Agent 执行规则。
- 工具介绍可以变成可选附录，只保留“什么时候使用”的条件。
- Memex、哲学来源、说服性段落可以从执行规则中删除。
- 漂亮但不可执行的比喻可以保留在 README，不应作为规则主体。

## 建议的文档分工

可以把项目中的文档拆成两类：

1. `README.md`

   面向人类读者，解释这个项目是什么、为什么有用、可以如何使用。

2. `AGENTS.md`

   面向 Agent，规定目录结构、操作流程、更新规则、页面模板和质量检查标准。

这样 `llm-wiki.md` 中的理念性内容不会丢失，但 Agent 执行时也不会被大量叙述性文字干扰。

## 改写方向

如果继续推进这个项目，可以把 `llm-wiki.md` 视为原始理念稿，然后生成一份更短、更硬的 `AGENTS.md`。

它的语气应从“这个想法是什么”改成“你在维护 wiki 时必须怎么做”。重点不是说服 Agent，而是约束 Agent 的行为。
